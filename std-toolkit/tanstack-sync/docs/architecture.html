How it all fits together

One collection pulls in a lot of concepts — strategies, partitions, a source of truth, per-partition sync state, a parallel cadence repair loop. This page is the mental model that ties them together, traced as five layers from "a UI mounts a query" down to "rows appear, stay fresh, and survive reload."

The whole system in one sentence: a collection is one Shape; a keyed collection runs sibling Partitions gated by subscriber count; inside each live partition a Strategy and a Cadence loop run in parallel, both writing one shared Source of Truth while each partition keeps its own detached Sync State; a Projector turns SoT into the UI's rows.

1 Shape — what a collection is

Every collection is one of two shapes. The shape decides whether partitions even exist.

singleItemSync

Single-item

Holds exactly one record, no id field, no partitions. Lifecycle is collection-level. A degenerate case of everything below: one runtime, one record.

sync

Keyed

A keyed set of records. Declares an optional global strategy and/or a flat map of field-value partitions. Both coexist over one shared SoT.

Within a keyed collection there are two kinds of partition, and they are siblings:

  • The global partition (internal key __total__) — mirrors the whole set. Starts when the collection's sync starts.
  • Field-value partitions — one per partitionField = value slice, e.g. status = "open". Activated on demand when a UI query loads that subset.
⚠️

Myth: partitions nest inside the "total" partition

They don't. The global partition is not a container — it is a peer that happens to cover the whole set. Global and field partitions are flat siblings sharing one SoT id-space; convergence by _u dedupes any row that two partitions both deliver. There is no recursion: a strategy is partition-blind and cannot spawn sub-partitions. "Total vs individual" is about which slice a runtime covers, never nesting.

2 Lifecycle — when sync runs, driven by subscriber count

A partition is a lifecycle boundary, not a retention boundary. The engine refcounts how many UI subscribers want each slice. The refcount edges — not data, not timers — start and stop work.

Edge What fires Tracked by
0 → 1 Open a scope; fork the strategy fiber + cadence fiber global: TanStack subscriberCount · field: engine refcount via loadSubset
1 → 0 Close the scope → both fibers torn down via their finalizers same, via unloadSubset
📌

Stopping sync ≠ removing rows

When a partition's refcount hits 0 the engine stops sync work (fibers, fetches, subscriptions). Rows that partition already projected stay visible in the collection. The scope is the unit of teardown; the projected rows are not.

3 Runtime — what runs inside one live partition

Once a partition's scope is open, two fibers run in parallel under it. They never coordinate directly — they meet only at the shared Source of Truth.

Strategy fiber

Forward progress

Partition-blind. Given its cursor, it pulls or streams the next batch of server entities and writes them to SoT, then advances its own Sync State. Decides the order data flows in (oldest-first, newest-first, both ends).

Cadence fiber

Drift repair

Scans the partition's projected rows for suspects — records delivered so close to their own _u that siblings may still be in flight. After a readiness window it re-fetches from a predecessor anchor and writes SoT. Repairs gaps the strategy can't see.

🔀

Cadence is repair, not a second strategy

The cadence loop never touches Sync State — it owns no cursor and makes no forward progress claim. It only re-confirms recent rows into SoT so the strategy's "covered" ranges don't quietly drift. It is optional per partition (cadence?: CadenceConfig | false) and inherits a default from the factory.

4 Shared state — two stores that never derive from each other

Source of Truth

Per collection · one id-space

Server-confirmed entities { value, meta }, keyed by id, tombstones kept. Written by strategies, mutations, and persisted broadcasts. Atomic batches; convergence keeps the newest _u. Shared by all partitions of the collection.

Sync State

Per strategy · per partition

One serializable slot of strategy bookkeeping — a cursor, or a list of covered _u slices. Wrapped on disk as { strategy, value, meta }; a name/schema mismatch resets it to empty. Carries no server truth.

🧩

The detachment rule (the load-bearing invariant)

SoT and Sync State are fully detached — neither is ever derived from the other. The engine never reads SoT to reconstruct a cursor; a strategy never reads its slices to decide what truth to keep. A strategy always writes SoT first via writeServerTruth, then advances its own state. This is why a swapped or reset strategy must re-fetch rather than "rebuild from SoT," and why durable offline storage can persist both independently.

5 Projection — SoT becomes the rows the UI reads

The Projector is the only writer to the TanStack DB collection. It translates SoT deltas into insert/update/delete messages: newer _u updates a row, a _d: true tombstone removes the visible row. Projection is ephemeral — it clears on reload and is rebuilt from persisted SoT when durable offline storage is configured.

👁️

What the UI sees vs. what is true

The collection (Items: flat fields + _meta) is a projection, not the truth. SoT can be written while the collection is unmounted; on mount the engine projects current SoT in. The Inspector reads the layers below the projection — partition activity, strategy progress, storage health — for devtools.

↳ Strategy family — the only thing that varies the order of forward progress

All keyed strategies share one contract — run(ctx) with writeServerTruth, getState/setState, and a scope. They differ only in how they walk _u. Their soundness rests on the Upward Migration invariant: an edit moves a record's _u strictly to "now," so a fully-drained _u interval is permanently complete.

Strategy Shape Sync State Use when
oldToNew Single forward pull, oldest → newest. Completes when a batch drains. { cursor } Small/medium backlog; newest rows can wait for replay.
newToOld Live tail at the fresh top ∥ one descending backfill frontier. { slices[], reachedOldest } Large backlog; newest rows must appear immediately.
bidirectional Live tail ∥ descending frontier ∥ ascending frontier; gap closes from both ends. { slices[] } Both newest and oldest must appear immediately.
Single-item family: getOnce fetch once on start · poll repeat while active · subscribe stream while active.

See ADR 0003 for why newToOld tracks slices, and CONTEXT.md for the full ubiquitous language.
