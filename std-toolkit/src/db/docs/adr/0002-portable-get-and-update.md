# getAndUpdate is the portable update; DynamoDB alone keeps a native update

Every entity service (multi- and single-entity, all three adapters) exposes `getAndUpdate`/`getAndUpdateOp`: read the current entity, derive a partial from it (plain partial or `current => partial | null` callback over the domain value — `null` skips the write), and write the full merged record back guarded on the `_u` that was read. On conflict, `getAndUpdate` re-reads and re-runs up to `retries` times (default 3) before surfacing `conditionFailed`; `getAndUpdateOp` cannot retry because its guard fires later inside `transact`. `lastWriteWins: true` drops the guard, accepting a full-record read-time-merge clobber — uniformly on every adapter, so the conformance suite pins one semantic. SQLite and IndexedDB renamed their `update`/`updateOp` to this (their updates were already read-modify-write internally; the old names hid the read and, outside transactions, carried no guard at all). DynamoDB instead carries two surfaces: the native `update`/`updateOp` (expression builders, custom conditions, autoMigrate, single round-trip) plus the portable pair.

The trade-off is deliberate: `getAndUpdate` costs two round-trips on DynamoDB where the native `update` costs one. Portability, the concurrency guard, and callback-over-current-value ergonomics won for cross-adapter code; latency-sensitive Dynamo-only code keeps the native path. Single-entity `getAndUpdate` treats the schema default (sentinel `_u: ''`) as a valid current value and guards the write on "record does not exist yet", so `noItemToUpdate` is not part of that path. Single-entity `updateOp` still fails with `noItemToUpdate` before the first persisted write.

Rejected: an auto-retry-free variant where callers hand-roll the retry loop (every caller writes the same loop); field-level patch writes under `lastWriteWins` (subtly adapter-dependent clobber scope, unconformable); keeping `update` alongside `getAndUpdate` on SQLite/IndexedDB (two names for one behavior).
