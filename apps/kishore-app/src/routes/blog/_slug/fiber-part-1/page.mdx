## What is a Fiber?

A Fiber is a lightweight, concurrent unit of execution that represents a running instance of an Effect.

Okay, that was a mouthful. Let's break it down.

Here is a simple illustration showing the relationship between Effect, Effect Runtime, and Fiber.

<OverviewImage />

An Effect represents a description of a computation. While a Fiber represents the running instance of that computation.

Let's see how they connect together.

### 1. Effect: (Definition of computation)

Effect as a description perfectly aligns with functions.

Just like we define a function, we define an Effect.

Defining does nothing in both cases. It just creates a value that describes what to do when executed.

```ts
import { Effect } from 'Effect';
function fn() {
  console.log('Hello, World!');
}

const effect = Effect.log('Hello, World!');
```

Here, neither `fn` nor `effect` do anything.
They only describe what to do when executed.

### 2. Execution

Functions are first-class in JavaScript. JavaScript knows what a function is, and how to execute it.

Running a function is as simple as calling it. `fn()`.

But Effect? JavaScript has no idea what it is. It is an abstraction created by the Effect.TS library.
So, how do we run an Effect?

We use the Effect itself to execute an Effect.

```ts
import { Effect } from 'effect';

const effect = Effect.log('Hello, World!');

// This is the key. Effect.runSync uses Effect runtime to execute the Effect.
Effect.runSync(effect);
```

We talked about defining and executing both functions and Effects.
Both had similarities in this regard: defining and executing.

Next comes the difference.

### 3. Fiber: (Running instance of computation)

Once you execute a function, there is no way you can control it. JavaScript does not provide any reference to a running execution. A function is a fire-and-forget operation.

But in Effect.TS, when you run an Effect, it creates a Fiber that represents the running instance of that Effect.

Now that we have a fiber, if we get the handle of it, we can control the running instance!

<CodeBlock id="interrupt" />

A fiber does not always run until completion. It is guided by a few rules to make concurrency safe and predictable.
That brings us to our next topic.

## Structured Concurrency

Effect enforces structured concurrency using Fibers.

So, what is structured concurrency?
Structured concurrency is a programming paradigm that ensures that concurrent tasks are organized in a hierarchical manner, where parent tasks manage the lifecycle of their child tasks.

In our case: fibers. A parent Fiber can fork child Fibers to run concurrent tasks. However, the lifecycle of these child Fibers is tied to the parent Fiber.

No child fiber can outlive its parent fiber. This ensures that the program's behavior is predictable and easier to reason about.

When you fork child Effects from within a parent Effect, the runtime ensures that the child Fiber's lifecycle is tied to the parent Fiber.

<CodeBlock id="structured-concurrency" config={{ enableTooltip: true }} />

The parent (**structured-concurrency-demo**) ran for a total of **1.5 seconds**.

- Fork returns immediately, so the time taken to fork children does not add to the parent's run time.
- The only time spent is 1.5 seconds sleeping.

We forked 3 child effects in the parent effect: _forked-child:1_, _forked-child:2_, _forked-child:3_.

In the above example:

1. ✅ **forked-child:1** ran for **1 second**.
   - Because forked-child:1 only needed 1 second, it ran successfully.
2. ⚠️ **forked-child:2** ran only for **1.5 seconds** and then got interrupted.
   - Why? That's structured concurrency in action! forked-child:2 takes 2 seconds to execute. But the parent only ran for 1.5 seconds. When parent completes, all its children are interrupted. So, forked-child:2 got interrupted after 1.5 seconds.
3. ❌ **forked-child:3** never got a chance to even run.
   - Why? Because it was the final operation in the parent Effect, even though forked-child:3 was forked, it was never executed because the parent Effect completed after 1.5 seconds.

> That's structured concurrency in action! Child Effects cannot outlive their parent Effect.

Structued concurrency is the default behaviour. We can always outlive the parent Effect by forking the child as a daemon, or using scope to manage the lifecycle manually.

But by default, Effect enforces structured concurrency to make concurrent programming safe and predictable.
