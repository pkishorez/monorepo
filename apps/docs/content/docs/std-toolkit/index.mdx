## Single-table design

Picture one table. Not one table per entity — *one* table for everything: tasks, users, settings, orgs, all of it as rows side by side.

How do you tell the records apart? Two mechanisms, working together:

- **By entity.** Every record carries its entity type, so a task row and a profile row can share a table without ever being confused for one another.
- **By partition.** Every record also carries a **partition key** that decides where it lives, and a **sort key** that orders records within that partition. Records that share a partition key sit physically together, in sort-key order.

That second group — all the records behind one partition key — has a name: an **item collection**. It is the natural unit of access in this design. "Everything about org `acme`" is not a join across five tables; it is one item collection, read in one query.

This is single-table design, and it is one of the great tricks of backend data modeling.

## Why DynamoDB makes it awesome

DynamoDB is where this design was born, and DynamoDB rewards it extravagantly: **any operation inside a single partition is guaranteed single-digit-millisecond latency** — at any scale, whether the table holds a thousand rows or ten billion. The partition is the unit of locality; keep your access patterns inside it and the database simply cannot be slow.

Model your domain as item collections and every hot path in your application becomes one predictable, fast partition read.

## The inspiration

Meanwhile, on the frontend, TanStack DB solved a different problem: keeping a live, queryable collection of records in the browser, with optimistic writes and reactive queries.

Put the two side by side and an idea falls out: **an item collection on the server is exactly the thing a frontend collection wants to mirror.** Both are sorted sets of items with a stable identity.

That is the inspiration for std-toolkit: an **end-to-end sync engine between DynamoDB and TanStack DB.** On the server, your data lives as single-table item collections. On the client, a `sync` abstraction mirrors one item collection into a TanStack DB collection and keeps it converged — with local (offline) storage so the mirror survives a reload.

If the server models everything as item collections, syncing to the client stops being a per-endpoint, per-feature chore — it becomes one generic mechanism.

## Why stop at DynamoDB?

This is all great — but why should only DynamoDB users reap these benefits?

It turns out the principle is generic. All the sync engine ever asks of the database is: *give me the sorted items of one item collection.* Any store that can hand back sorted rows behind a key can play the part — including SQL databases. std-toolkit ships a SQLite adapter that speaks the exact same partition-key / sort-key / item-collection topology.

Yes, you give up the power of SQL: no ad-hoc joins, no arbitrary queries against your synced data. But the disciplined single-table approach buys things SQL cannot give you here — the same data model in production DynamoDB and in tests, local dev, Bun, or a Cloudflare Durable Object, with zero mocking; access patterns designed up front instead of discovered in a slow-query log; and one sync mechanism that works unchanged over any backend.

That is std-toolkit in one line: **database-agnostic sync over single-table item collections.**

One more problem hides inside this design: single-table stores are schemaless, and long-lived synced data *will* change shape. std-toolkit answers that with [evolving schemas](/docs/std-toolkit/eschema).
