**std** stands for **single-table design**.

Modern apps want data to feel local.

A screen should open with data already there. Changes should appear immediately. The app should keep working offline and catch up when it reconnects.

The difficult part is not storing data. It is keeping the right data on every client correct and up to date.

## The usual sync problem

Application data is spread across tables and connected by relationships. A channel screen, for example, may need a channel, its messages, and its members.

Most sync engines discover changes by watching the database's replication log. That gives them a stream of rows, but now the engine must understand the database, its tables, and how those rows relate to what each client needs.

This works, but it couples sync to the database's internals. Change the database and the sync engine may no longer work. Use a database without the expected replication stream and it cannot start.

std-toolkit begins with a smaller question:

> What is the least a sync engine needs from a database?

The answer is not tables, joins, or a replication log.

It needs **items that can be read in order**.

## The idea comes from DynamoDB

This is not a new storage model. std-toolkit is directly inspired by **DynamoDB single-table design**.

DynamoDB stores items in one table using two keys:

- a **partition key** groups related items;
- a **sort key** orders the items inside that group.

DynamoDB calls the items behind one partition key an **item collection**. A Query selects one collection and reads a sorted range from it.

That is the important insight std-toolkit borrows: one physical table can contain many independently addressable, ordered collections.

std-toolkit takes that DynamoDB model and treats it as a portable database contract. DynamoDB implements it natively. SQLite can implement the same shape with a table and composite indexes. The sync layer only sees item collections, so it does not need to understand which database produced them.

## The small primitives

std-toolkit builds the whole model from four primitives.

### Entity

An entity is a typed piece of application data: a `Channel`, `Message`, or `Member`.

Each entity defines its own shape, identity, and schema history. Different entities remain different even though they share one physical table.

### Item collection

An item collection is a group of one entity type, in a stable order.

For a chat app, useful collections include:

```text
Channel
Message#general
Message#random
Member#general
Member#random
```

`Message#general` contains only messages from `#general`. `Member#general` contains only its members.

The partition key chooses the collection. The sort key orders its items. Together, those two keys turn one physical table into many small collections.

### Cursor

A cursor is a client's position in a collection.

Once a collection is ordered, sync becomes a range query:

> Give me the next items in `Message#general` after this cursor.

The client persists the cursor and can resume after a reload, a lost connection, or a week offline. The server does not need to remember where every client stopped.

### Transaction

A transaction applies operations to several items as one unit.

The items do not need to belong to the same entity. Creating a channel, its owner membership, and its settings can be one transaction: either every write succeeds or none does.

The server does not join entities before sending them. Each query returns one predictable, typed collection.

On the client, std-toolkit stands on **TanStack DB**. TanStack DB provides the typed local collections, reactive live queries, client-side joins, and optimistic mutations that make local-first screens practical.

std-toolkit's [`tanstack-sync`](/docs/std-toolkit/tanstack-sync) package connects the two sides. It drains server item collections into TanStack DB collections, keeps their sync progress, and converges newer writes over older ones.

The channel screen can now join its local channel, message, and member collections. A different screen can combine the same collections differently without changing the server's storage model or sync protocol.

This is the key shift: **the server stores and syncs entities; the client composes them into screens.**

## What about relationships?

A relational database can enforce that every message points to an existing channel with a foreign key. A single-table item model cannot express that relationship to the database.

The application owns it instead.

For example, sending a message can be one transaction that first checks the channel exists and then writes the message. The transaction guarantees that the check and the write succeed together.

But a transaction does not invent that rule. Another code path could still write a message without checking the channel unless the application consistently enforces the invariant.

So the boundary is precise:

- **transactions provide atomicity** — all related item operations succeed or none do;
- **application code provides structural integrity** — it decides which entities may relate and keeps those rules true.

That is the tradeoff. std-toolkit gives up database-enforced joins, foreign keys, and cascades. In return, every backend only needs to implement the same small item-collection operations.

## Why this is enough for sync

A sync engine does not need to understand the entire database. It only needs to move changes for the collections a client cares about.

std-toolkit gives every write an ordered update key. Inserts, edits, and tombstones can therefore be read as an ordered timeline. The client keeps the newer copy of an entity and ignores older or duplicate deliveries.

From those rules, the rest follows:

- initial sync starts at the beginning of a collection;
- pagination continues from a cursor;
- reconnect resumes from the saved cursor;
- live delivery feeds the same local collections;
- duplicate and out-of-order deliveries converge safely;
- the client syncs only the collections its current views need.

DynamoDB can answer these ordered queries directly. SQLite can answer them with a composite index. Neither database needs to expose a replication log to std-toolkit.

## Standing on the shoulders of giants

std-toolkit exists because two larger ideas fit together remarkably well.

**DynamoDB** provides the server-side inspiration: one table, partition keys, sort keys, item collections, and ordered range queries.

**TanStack DB** provides the client-side foundation: typed local collections, reactive queries, joins, and optimistic state.

std-toolkit is the bridge between them. It turns DynamoDB's item-collection model into a database-agnostic sync contract, then feeds those collections into TanStack DB on the client.

That is the whole insight:

> **One table holds the entities. Item collections make them syncable. The client turns them back into an application.**
