## The itch

DynamoDB's single-table design is one of the great tricks of backend modeling: instead of one table per entity, you store *every* entity type in one table. A **partition key** decides where a row lives, a **sort key** orders rows within that partition, and the set of rows sharing one partition key — the **item collection** — becomes the natural unit of access. One query, one partition, everything you need.

Meanwhile, on the frontend, TanStack DB solved a different problem: keeping a live, queryable collection of records in the browser, with optimistic writes and reactive queries.

Put the two side by side and an idea falls out: **an item collection on the server is exactly the thing a frontend collection wants to mirror.** Both are sorted sets of items with a stable identity. If the server models everything as single-table item collections, syncing to the client stops being a per-endpoint, per-feature chore — it becomes one generic mechanism.

That is std-toolkit: **database-agnostic sync over single-table item collections.** Thanks to DynamoDB for the data model and TanStack DB for the client runtime — std-toolkit is the bridge between them.

Each item collection on the left has a stable identity (its partition key) and a stable order (its sort key). That is *all* the sync engine needs — it does not care whether the table is DynamoDB, SQLite, or anything else that can hand back sorted items.

## Why "database-agnostic" matters

The sync layer only ever asks for sorted items out of an item collection. So the same application model runs against:

- **DynamoDB** in production — the pattern's birthplace.
- **SQLite** in tests, local dev, Bun, or a Cloudflare Durable Object — same topology, same vocabulary, zero mocking.

One more problem hides inside this design: single-table stores are schemaless, and long-lived synced data *will* change shape. std-toolkit answers that with [evolving schemas](/docs/std-toolkit/eschema) — versioned schemas that migrate old data forward on read.
