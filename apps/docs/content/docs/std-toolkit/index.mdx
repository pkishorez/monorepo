You already know the experience you're after. You're building a chat app: someone opens a channel and the messages are just *there* — live queries against local data, optimistic sends that feel instant, updates from other people streaming in, everything surviving a page reload. You've seen apps that work this way, and you've probably scoped out what it would take to build one.

And that's where it gets uncomfortable, because every road to that experience asks you to give something up:

- **Sync engines that own your database.** The most polished ones work by tapping the database's replication stream — they sit on the write-ahead log or a change-data-capture feed and fan changes out to clients. Beautiful, until your data lives somewhere that doesn't expose a replication stream in the shape they need, or somewhere you don't fully control. The engine doesn't adapt to your database; your database choice is dictated by the engine.
- **Whole-database replication to the client.** Ship the database — or a large slice of it — into the browser and query it locally. Now you've adopted a specific storage engine on both ends, you're syncing far more than any one screen needs, and partial replication becomes its own research project.
- **Hand-rolled fetch + cache invalidation.** The default. You write the endpoints, the paging, the cache keys, the invalidation rules, the reconnect logic, the "wait, why is this message duplicated" fix, the "wait, why did this deleted message come back" fix. Every feature adds another edge case, and none of it composes.

So the hard question: what is the *smallest* thing a sync engine could demand from a database and still deliver that experience?

## The one idea

std-toolkit's answer: **sorted items**. That's the entire ask.

Take your chat app. Give every message a partition key (the channel it belongs to) and a sort key (a time-ordered ULID). All the messages in one channel now form one **item collection** — a set of items behind one partition key, sorted by the sort key. To sync that collection, an engine only ever needs to ask one question:

> "Give me the items in this collection, in order, after this point."

Any database that can answer that question can be synced. DynamoDB answers it natively — it's literally what a Query does. SQLite answers it with `WHERE pk = ? AND sk > ? ORDER BY sk`. No replication stream, no change feed, no storage engine embedded in the browser. That's why std-toolkit runs over DynamoDB, where replication-stream-based engines can't follow you, and over SQLite with the same code — and why nothing about it is married to either.

Everything in the toolkit is built outward from this primitive:

- Schemas version themselves, because rows written last year live in the same sorted collection as rows written today.
- The database adapters exist to derive keys and maintain item collections, not to be ORMs.
- The sync engine's entire persistent state per collection is, essentially, a cursor into a sorted list.

The mental model comes from DynamoDB's single-table design — with one deliberate twist (collections hold a single entity type) that we'll defend honestly when you get to [item collections](/docs/std-toolkit/item-collections).

## One thing to know before the code starts

std-toolkit is an [Effect](https://effect.website)-native toolkit, end to end. Errors are typed, dependencies are Layers, and every operation is an `Effect`. The docs assume you're fluent — code samples use `Effect.gen`, `yield*`, and services without re-explanation. If you're evaluating std-toolkit, you're almost certainly evaluating Effect too; that's a package deal, and we'd rather say so on page one.

Here's the destination, so you know where this is all heading — the browser side of the chat app, where opening a channel activates sync for exactly that channel's item collection:

```ts title="collections.ts"
const messages = createCollection(
  std.sync({
    schema: Message,
    sync: {
      partitions: {
        channelId: (channelId) => ({
          strategy: syncStrategy.oldToNew(),
          forwardFetch: ({ cursor }) => api.messagesAfter(channelId, cursor),
        }),
      },
    },
    onInsert: (message) => api.sendMessage(message),
  }),
);
```

One config: which collection, how to page through it, what to do on a write. The engine handles cursors, convergence, deduplication, tombstones, offline storage, and reconnect — because the primitive underneath is simple enough that it *can*.

## The map

The toolkit is a small stack of subpaths, each one layer of the same story:

1. **[eschema](/docs/std-toolkit/eschema)** — versioned, self-migrating schemas. Your message schema will change; the rows already written won't. An eschema always encodes the latest version and folds old rows forward on decode.
2. **[core](/docs/std-toolkit/core)** — the Entity envelope. Every layer speaks `{ value, meta }`, where `meta` carries the schema version, entity type, tombstone flag, and the update key that makes convergence a one-line comparison.
3. **[Database adapters](/docs/std-toolkit/database-adapters)** — [DynamoDB](/docs/std-toolkit/dynamodb) and [SQLite](/docs/std-toolkit/sqlite) services that put entities into item collections and query them back, with type-safe key derivation.
4. **[tanstack-sync](/docs/std-toolkit/tanstack-sync)** — the browser end: TanStack DB collections fed by sync strategies that drain item collections through cursors, with offline storage and paced optimistic writes.
