---
title: Item Collections
description: The small, sorted, typed unit that std-toolkit queries, paginates, and syncs.
---

The [DynamoDB crash course](/docs/std-toolkit/dynamodb-crash-course) gave us one useful picture: a table is divided by partition keys, and a sort key gives every item inside a partition a stable position.

All the items behind one partition-key value form an **item collection**.

std-toolkit takes that DynamoDB concept and makes it the boundary for querying and sync. The backing store may be DynamoDB or SQLite; the rest of the toolkit only needs to see small, ordered collections.

## The table is the home, not the working set

One application table can contain many collections:

```text
application table
├── Channel
├── Message#general
├── Message#random
├── Member#general
└── Member#random
```

The table is where all entity types live. A collection is the slice the application works with right now.

Opening `#general` does not mean reading every message in the table. It means addressing `Message#general`. Opening its member list means addressing `Member#general` separately.

One table is therefore not one download. It is a home for many independently readable pieces.

## A useful collection is addressable

The partition key names the collection precisely. A query for `Message#general` does not search other channels to discover which messages belong there.

The grouping is chosen from the way the application reads. Because a chat screen needs one channel's messages together, the channel id becomes part of the message partition key.

This keeps each request focused on the data needed by one feature or view.

## A useful collection is ordered

The sort key gives every item a stable position inside its collection.

That makes pagination a continuation instead of a search. Read a page, remember the last sort key, and ask for the items after it.

```text
server:  A  B  C  D  E  F
client:  A  B  C
cursor:        C
next:             D  E  F
```

If the browser closes after `C`, it persists `C`. When it returns, it continues from there. The server does not need to keep a sync session for that browser because the client carries its own progress.

The cursor is small, requests can be repeated safely, and recovery does not require comparing whole database snapshots.

## Creation order and change order are different

A primary collection is often ordered by entity id. With ULID ids, `Message#general` naturally reads in creation order.

That order is ideal for chat history. It can show the oldest messages, the newest messages, or the next page after a visible message.

But an edit does not change a message's id. A client that only continues after the latest message id would never revisit an older message that was edited.

std-toolkit gives every write a fresh `_u` update key. A secondary index can order the same collection by `_u`, so inserts, edits, and tombstones all move through one change timeline.

```text
same message
├── messageId  → its place in chat history
└── _u         → its place in change history
```

The primary order answers “where was this created?” The change order answers “what changed since I was here?” A complete sync model needs to understand both jobs.

## std-toolkit keeps collections typed

Classic DynamoDB single-table design often mixes several entity types behind one partition key:

```text
Channel#general
├── META
├── MESSAGE#01J...A1
└── MEMBER#user-a
```

One Query can then return a channel, its messages, and its members together. This can be a useful DynamoDB pattern, but it produces a heterogeneous result.

std-toolkit makes a different tradeoff. It includes the entity type in the partition key and keeps each collection homogeneous:

```text
Channel
└── general

Message#general
└── 01J...A1

Member#general
└── user-a
```

Now a message query returns `Message[]`, not `(Channel | Message | Member)[]`. Every item is decoded by the same eschema, uses the same identity rule, and follows the same migration and convergence rules.

The sync engine receives data in the shape the client ultimately needs: one typed collection at a time.

## Relationships remain explicit

A channel screen still needs channels, messages, and members. std-toolkit reads those typed collections independently and lets application code or TanStack DB combine them.

That costs additional queries, but the queries can run concurrently. In return, relationships remain visible in types and code instead of being hidden inside key prefixes.

This separation only applies to reads. A transaction can still write a channel, its owner membership, and its settings atomically, even though they belong to different entity types and collections.

The model is: **reads stay typed; related writes can stay atomic.**

## The collection becomes a database contract

Once the application is expressed as item collections, std-toolkit needs very little from a database:

1. select a collection by partition key;
2. return its items in sort-key order;
3. continue before or after a cursor.

DynamoDB provides that shape directly with Query. SQLite can provide the same shape with a composite index:

```sql
SELECT * FROM items
WHERE pk = ? AND sk > ?
ORDER BY sk
LIMIT ?;
```

The implementation is different, but the question is identical:

> Give me the next items in this collection after this cursor.

That is why item collections are enough to make the sync layer database-agnostic. The sync engine does not need tables, joins, or a database-specific replication log. It needs a named group, a stable order, and a position it can resume from.

## The mental model so far

DynamoDB gave us one table with direct, ordered access paths. std-toolkit narrows those paths into typed item collections. Cursors turn their order into resumable progress, and a change-order index makes edits and deletions visible as well as inserts.

We now have the storage and movement primitives. The last step is to follow a real message through every layer and see how those primitives become an application.
