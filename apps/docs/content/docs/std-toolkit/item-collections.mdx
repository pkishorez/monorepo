title: Item collections
description: Why a sorted, single-entity-type item collection is everything a sync engine needs — and why std-toolkit breaks with classic single-table design to get it.
---

You now speak DynamoDB: partition key, sort key, and the [item collection](/docs/std-toolkit/dynamodb-crash-course) — all items behind one partition key value, in sort-key order. DynamoDB treats the item collection as one concept among several. std-toolkit strips everything else away and keeps *only* this one. Every part of the toolkit — the schema layer, both database adapters, the sync engine — is built against the same primitive:

> **An item collection: a sorted set of items behind one partition key, all of the same entity type.**

In the chat app: one channel's messages. `pk = channel id`, `sk = message ulid`, every item a message. That last clause — *all of the same entity type* — is where std-toolkit deliberately breaks with classic DynamoDB modeling, and we'll defend that honestly in a moment. First, the payoff.

## All a sync engine needs

Think about what it takes to mirror one channel's messages into a browser, correctly, forever. You need initial load, catch-up after being offline, live updates, and you need it all to be resumable — a browser tab can die at any moment.

Now notice what the item collection gives you for free. The collection is *sorted*, and the sort key is a stable cursor into it. So the entire protocol collapses to one question, asked repeatedly:

> "Give me the items in this collection, in order, after this cursor."

- **Initial load?** Ask with an empty cursor, keep asking until the answers run dry.
- **Catch-up?** Ask with the last cursor you saw.
- **Resume after a crash?** Persist the cursor; ask again from there.
- **Live updates?** New messages get new, larger sort keys — they show up after your cursor by construction.

No server-side session, no change feed, no replication log, no per-client bookkeeping on the server at all. The sync state is a cursor, and the cursor belongs to the client.

And here's why std-toolkit calls itself database-agnostic: that question is answerable by almost anything. DynamoDB answers it natively:

```
Query  pk = "channel#general", sk > "01HZX3GA91..."
```

SQLite answers it with one indexed query:

```sql
SELECT * FROM items
WHERE pk = ? AND sk > ?
ORDER BY sk
LIMIT ?;
```

Any store that can return sorted items after a key can sit under std-toolkit's sync engine. The engines that need to own your database's replication stream demand something rare; this demands something nearly every database already does.

## The heresy: one entity type per collection

If you've read anything about DynamoDB single-table design, you know its signature move: *mix entity types in one partition*. The classic pattern puts a channel and its messages in the same item collection, discriminated by sort-key prefixes:

| pk                | sk               | ...                              |
| ----------------- | ---------------- | -------------------------------- |
| `channel#general` | `META`           | `{ name: "general", topic: … }`  |
| `channel#general` | `MSG#01HZX3F8B2` | `{ body: "morning all", … }`     |
| `channel#general` | `MSG#01HZX3GA91` | `{ body: "anyone seen…", … }`    |

One Query now returns the channel *and* its messages — a "join" with a single round trip. It's genuinely elegant, it's the technique the whole single-table literature is built around, and std-toolkit doesn't do it. In std-toolkit, item collections are homogeneous: a collection of messages is messages, all the way down.

You should be suspicious of a toolkit that names itself after single-table design and then drops the pattern's most famous trick. Here are the three honest reasons.

**1. Typing a heterogeneous Query result is a mess.** That mixed Query returns `(Channel | Message)[]` — except the type system doesn't know that. Each row must be routed to the right schema by inspecting its key string, every consumer narrows the union by hand, and validation can't run against one schema for the whole result. In a TypeScript codebase, the exact place you most want precise types — the data crossing your database boundary — is where this pattern makes them weakest.

**2. The relationships are conventions, nothing more.** Nothing in the table declares that `MSG#` rows belong to the `META` row. The relationship lives in key-string prefixes and in your team's heads. There's no schema to enforce it, no type to represent it, and no principled answer for how deep it goes (do reactions nest under messages under channels?). Heterogeneous partitions make relationships *implicit* exactly where you'd want them explicit.

**3. The sync engine deals with one entity type at a time — and this is the decisive one.** Walk the browser end backwards: the thing you render from is a typed collection of messages. Convergence compares update keys per message; tombstones delete messages; schema migrations fold old message rows forward through *the message schema*. Every one of those mechanisms is defined per entity type, and every synced envelope is tagged with its type so the engine can reject strays. If the wire carried heterogeneous partitions, the very first thing the sync engine would have to do is split them back apart by type. The mixed collection would exist only to be un-mixed. So std-toolkit un-mixes it at the source: the partition key *includes the entity type*, and each type gets its own item collections.

<Callout title="Transactions are not the thing being given up">
Cross-entity *queries* are excluded; cross-entity *transactions* are not. The `EntityRegistry` coordinates every entity type in a table and exposes a typed `transact` that accepts operations from any of them — creating a channel and its first message atomically is one call. What you can't do is *read* two entity types in one query. Writes coordinate; reads stay homogeneous.
</Callout>

## Recomposition: the join moves to the client

So how do you render a channel screen, which needs the channel *and* its recent messages? You ask two questions and join in code — and because the queries are independent, you ask them concurrently:

```ts title="channel-page.ts"
const channelPage = (channelId: string) =>
  Effect.gen(function* () {
    const channels = registry.entity('channel');
    const messages = registry.entity('message');

    const [channel, recent] = yield* Effect.all(
      [
        channels.get({ channelId }),
        messages.query(
          'primary',
          { pk: { channelId }, sk: { '<=': null } },
          { limit: 50 },
        ),
      ],
      { concurrency: 'unbounded' },
    );

    return { channel, messages: recent.items };
  });
```

Two Queries instead of one. Each hits exactly one item collection, each result decodes against exactly one schema, and the shape of the join is visible in your code instead of encoded in key prefixes. Yes, it's one more round trip than the classic pattern — round trips you fire in parallel, against a database where single-collection Queries are the cheap, fast case by design. That's the trade, stated plainly: std-toolkit pays a concurrent round trip to buy precise types, explicit relationships, and a sync engine that's even possible.

One vocabulary warning before you move on: when you reach the sync engine, you'll meet the word *partition* again — there it means a sync lifecycle window (an open channel being actively mirrored), not a storage slice. Same word, two contexts; the docs always make clear which one is in play.
