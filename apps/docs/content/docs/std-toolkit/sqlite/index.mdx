## The story

"Database-agnostic" is an easy claim and a hard property. The proof is a second database that runs the *same* model — not a lookalike API over a different mental model, but the same topology: one table, partition key, sort key, item collections, range queries within a partition.

`std-toolkit/sqlite` is that proof. It mirrors the DynamoDB adapter's topology over SQLite — the two share one storage kernel, so `SQLiteEntity` is `DynamoEntity` with a different engine underneath, and application code written against the entity services does not care which one it got. That buys you things DynamoDB alone cannot:

- **Tests without mocks.** Run the real storage layer in-memory or on disk; your test exercises the same single-table semantics as production.
- **Local dev with zero cloud.** No LocalStack, no credentials.
- **Runtimes DynamoDB can't reach.** SQLite runs in Node, Bun, and Cloudflare Durable Objects — each wired in through its own adapter subpath, so a Durable Object can hold a real item collection at the edge.

Because the sync layer only ever asks for sorted items out of an item collection, everything above this adapter — [eschema](/docs/std-toolkit/eschema) validation, [tanstack-sync](/docs/std-toolkit/tanstack-sync) mirroring — works identically over SQLite and DynamoDB. That is the toolkit's central bet, made concrete.
