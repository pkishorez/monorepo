You've built offline support before, and it probably looked like hydration: serialize some cache to localStorage on an interval, read it back on boot, hope the snapshot wasn't mid-write when the tab died. The hard part was never storing data — it's that a _snapshot_ of a sync system is a lie waiting to happen. The cache says one thing, the cursor says another, and the gap between them is invisible until a user asks where their message went.

tanstack-sync sidesteps the snapshot model entirely. Offline storage here is not a cache of the engine's state — it _is_ where the engine keeps its state.

## `idbStorage` — one line, durable engine

```ts title="client/std.ts"
import { createStdSync } from 'std-toolkit/tanstack-sync';
import { idbStorage } from 'std-toolkit/tanstack-sync/offline-storage/idb';

export const std = createStdSync({
  offlineStorage: idbStorage({ name: 'chat-app' }),
});
```

With this in place, the two stores from [truth and state](/docs/std-toolkit/tanstack-sync/truth-and-state) live in IndexedDB as their primary backend. Every Source-of-Truth write — a strategy batch, a mutation result, a persisted broadcast — lands in IndexedDB _as it happens_, inside the same write path that can fail and surface. Every Sync State commit does the same. There is no flush interval, no serialize-on-unload, and therefore no moment where memory and disk disagree about what's true.

When nothing is configured, the same interfaces run over an in-memory backend — identical semantics, gone on reload. Individual collections can override the instance default:

```ts
std.sync({
  schema: Draft,
  offlineStorage: idbStorage({ name: 'chat-drafts' }), // its own database
});

std.sync({
  schema: Presence,
  offlineStorage: false, // ephemeral by design — memory, even if the app is durable
});
```

`idbStorage` takes a `name` (the IndexedDB database) and an optional `version` for your own data-versioning. The factory is safe to call during SSR and module evaluation — validation is pure, and IndexedDB isn't touched until an actual storage operation runs in the browser.

### What a reload looks like now

Close the tab mid-backfill on a fifty-thousand-message channel. Reopen it. The collection projects straight from the persisted Source of Truth — messages on screen before any network round trip — and the partition's strategy reads its persisted [Sync State](/docs/std-toolkit/tanstack-sync/truth-and-state#sync-state) and resumes: `oldToNew` from its cursor, `newToOld` from its covered slices, no re-drain of what's already owned. The live tail catches up the gap since the tab died. Projection itself is rebuilt every boot — it's the one deliberately ephemeral layer, cheap to reconstruct from durable truth.

<Callout type="warn" title="Durable storage is not an offline mutation queue">
  Offline storage makes engine state survive reloads and network loss. It does
  not, by itself, define mutation queuing, conflict policy, or network behavior
  — a mutation fired while offline still fails through your Effect's error
  channel, and what to do about it (queue, retry, surface) remains your call.
</Callout>

## The `OfflineStorage` contract

IndexedDB is an adapter, not an assumption. Any backend implementing one interface can sit under the engine:

```ts
type OfflineStorage = {
  readonly descriptor: StorageDescriptor;
  group: (name: string) => OfflineStorageGroup; // named KV space: get/getAll/put/putMany/delete/clear
  clear: () => Effect.Effect<void, OfflineStorageError>;
  inspect: () => Effect.Effect<Array<{ group: string; entries: … }>, OfflineStorageError>;
};
```

A **group** is a named key-value space, and the engine's layout is two groups per collection: `sot/Message` (Source of Truth, keyed by entity id) and `state/Message` (Sync State, keyed by partition — one slot per channel, plus `__total__` for a global strategy). If you crack open DevTools → IndexedDB, that's exactly the structure you'll find; nothing about the engine's persistence is opaque.

`StorageDescriptor` is the storage's self-description — `{ kind: 'indexeddb', name: 'chat-app' }` or `{ kind: 'memory' }` — so tooling can address the underlying store (open the right database, label the right tab) without guessing what kind it is. New adapters add a variant rather than pretending to be an existing one.

## When storage fails

Durability adds a failure mode: the disk itself. A quota exceeded, a browser in a private window, a flaky IndexedDB — every storage operation returns a typed `OfflineStorageError`, and inside the engine's write path it surfaces as the `Storage` variant of `WriteError`:

```ts
{ _tag: 'Storage'; reason: string; cause?: unknown }
```

The important word is _surfaces_. A Source-of-Truth write that can't persist **fails** — atomically, nothing partial — rather than succeeding in memory and silently diverging from disk. From a strategy, that `WriteError` triggers the standard [recovery policy](/docs/std-toolkit/tanstack-sync/strategies#when-strategies-fail-the-recovery-policy): log, restart on the 2-second schedule, resume from the last state that actually committed. From `utils.writeUpsert`, it's in your Effect's error channel to handle. Either way, memory and disk cannot quietly disagree — the failure is loud at the moment it happens.

## The `SyncInspector` — seeing all of it

This section has described a lot of invisible machinery: partitions refcounting, strategies persisting slices, cadence loops idling. The last export is the window into it. Every `createStdSync` instance carries a read-only introspection surface:

```ts
const inspector: SyncInspector = std.inspector;
```

Its two main members are themselves live TanStack DB collections — which means you inspect the sync engine using the exact query machinery it syncs with:

```tsx title="devtools/sync-panel.tsx"
const { data: partitions } = useLiveQuery((q) =>
  q.from({ p: std.inspector.partitions }),
);
```

- **`inspector.collections`** — one `InspectorCollection` row per synced collection: its name, `kind` (an `InspectorCollectionKind`: `'keyed'`, `'partitioned'`, or `'single-item'`), live `status`, item and subscriber counts, and its declared partition fields.
- **`inspector.partitions`** — one `InspectorPartition` row per sync partition ever seen: which field and value it's scoped to, whether it's the `global` partition or a keyed one, its `activity` (`active` while subscribed, `cached` once unloaded — remember, [unload isn't forgetting](/docs/std-toolkit/tanstack-sync/collections#partitions--sync-lifecycle-windows)), item and subscriber counts, its resolved cadence config, and — the headline — its `strategyState`.

`InspectorStrategyState` is the strategy's persisted progress, decoded per strategy kind: `oldToNew` shows its cursor; `newToOld` its covered slices (with per-slice item counts) and `reachedOldest`; `bidirectional` its slices. It's the same structure the [visualizer](/docs/std-toolkit/tanstack-sync/strategies) renders in its Sync State readout — except here it's your app's real drains. On boot, partitions cached in offline storage are restored into the inspector, so the panel shows last session's coverage without waiting for a drain.

Around the two collections sits a small utility rim: `inspector.storage` exposes the root `OfflineStorage` (descriptor included, so devtools can name the database); `getCollection` fetches a synced collection by name; `readEntities` reads a collection's live persisted values, optionally narrowed to one partition; `countEntities` counts them; and two escape hatches — `clearEntries` (drop a collection's stored entities) and `clearSyncState` (reset one partition's drain progress so it re-syncs from scratch) — exist for the debugging sessions where you want to watch a drain happen again from zero.

That's the whole subsystem: one factory, three write families, three drain orders, two detached stores, one storage contract — and a window to verify every claim this section made against your own running app.
