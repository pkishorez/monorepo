## The story

Every app that shows server data in a UI ends up rebuilding the same machine: fetch a list, cache it, page it, patch it optimistically, reconcile the server's answer, survive a reload, don't resurrect things the user deleted. Usually that machine is rebuilt per feature — one for tasks, one for settings, one for notifications — each with its own subtle bugs.

The single-table model makes a better deal possible. If the server's unit of data is the [item collection](/docs/std-toolkit) — a sorted set of entities with a stable identity — then "show server data in the UI" has one general answer: **mirror the item collection into a TanStack DB collection, and keep the mirror converged.**

That is what `std-toolkit/tanstack-sync` does. You declare a collection with a schema and a **strategy** (how to pull items: old-to-new, paged, on demand), and the engine owns everything else:

- a **Source of Truth** per collection — server-confirmed entities only, atomically written, with tombstones retained so deleted data cannot come back;
- an ephemeral **collection projection** — the TanStack DB rows your components query live;
- durable **Sync State** — the strategy's cursor, so sync resumes where it left off after a reload.

Server-side partitions map naturally onto frontend life: a partition's strategy starts the first time a component queries for that slice (`loadSubset`) and stops when the last one leaves — while the rows it already delivered stay put. Optimistic writes go through paced mutation handlers; server broadcasts route through a registry by entity type tag. Offline storage (IndexedDB) is the live backend for truth and state, not a hydration-only cache.

One machine, written once, converged by construction — for every entity in your table.
