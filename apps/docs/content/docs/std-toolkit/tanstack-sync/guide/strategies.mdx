---
title: Sync strategies
description: oldToNew, newToOld, bidirectional, getOnce, and coalesce — what each is for and how they interplay.
---

A strategy answers one question: **in what order should the mirror converge on server truth?** Different UIs need different answers — a task list wants complete history, a chat wants the newest messages *now* — so the strategy is pluggable per collection (and per partition).

Every partitioned strategy runs against the same engine contract: it reads its own persisted **Sync State** (typically a cursor), fetches, writes batches through `writeServerTruth`, and advances its state. That is why a strategy survives page reloads and partition unloads — it resumes exactly where its cursor says it stopped.

## `syncStrategy.oldToNew` — drain history forward

The simplest shape: start at the oldest record and page forward until the set is drained.

```ts
sync: {
  strategy: syncStrategy.oldToNew(),
  forwardFetch: ({ cursor }) => client.queryTasks({ cursor }),
},
```

With no config, it polls `forwardFetch` batch by batch — each non-empty batch is written and the cursor advances to its newest entity; an empty batch means the drain is complete. Alternatively, pass a stream to keep the strategy alive as a subscription:

```ts
strategy: syncStrategy.oldToNew({
  stream: ({ cursor }) => client.streamTasks({ after: cursor }),
}),
```

**Use it when** the collection is small enough to mirror completely and the UI wants the full set — task boards, settings-like reference data, anything where "eventually complete, in order" is the right contract.

## `syncStrategy.newToOld` — newest first

For history that is deep but where the UI needs the *recent* end immediately — a message channel, an activity feed. Two sources, two jobs:

```ts
sync: {
  strategy: syncStrategy.newToOld({
    // finite: descends toward the oldest record, completes at the floor
    subscribeOlder: ({ cursor }) => client.streamOlderMessages({ cursor }),
    // live: starts strictly after the cursor, catches up, then stays open
    subscribeNewer: ({ cursor }) => client.streamNewerMessages({ cursor }),
  }),
  forwardFetch: ({ cursor }) => client.queryMessages({ cursor }),
},
```

The backfill descends into history while the live tail keeps the top fresh — the user sees today's messages instantly and yesterday's fill in behind them. Internally the strategy tracks *slices* (contiguous synced ranges) so an interrupted backfill resumes without gaps or re-fetching.

**Use it when** recency is what the user came for and full history is a background concern.

## `syncStrategy.bidirectional` — fill from both ends

Newest-first delivery that closes the backlog from **both** directions at once: `fetchOlder` descends from the newest end while the engine's `forwardFetch` ascends from the oldest, and they stop when the two synced ranges meet. A `subscribeNewer` live tail stays open throughout.

```ts
sync: {
  strategy: syncStrategy.bidirectional({
    fetchOlder: ({ cursor }) => client.queryOlderEvents({ cursor }),
    subscribeNewer: ({ cursor }) => client.streamNewerEvents({ cursor }),
  }),
  forwardFetch: ({ cursor }) => client.queryEvents({ cursor }),
},
```

**Use it when** you want newest-first UX *and* the fastest possible path to a complete mirror — the backlog closes in roughly half the time because both ends work simultaneously.

## `singleItemSyncStrategy.getOnce` — one record, one fetch

The single-item family has its own strategies. `getOnce` fetches the record once, writes it as server truth, and completes:

```ts
strategy: singleItemSyncStrategy.getOnce({
  get: () => client.getSettings(),
}),
```

**Use it for** singletons — settings, the current user's profile — where a one-shot read plus registry broadcasts covers freshness.

## `paceStrategy.coalesce` — pacing writes, not reads

Not a sync strategy but its outbound sibling: `updatePacing` controls how optimistic edits travel *to* the server. `paceStrategy.coalesce` merges rapid successive updates to the same key into one request, so a drag-resize or a fast typist doesn't produce a request storm:

```ts
std.sync({
  schema: TaskSchema,
  sync: { … },
  updatePacing: paceStrategy.coalesce({ wait: 50 }),
  onUpdate: ({ id, updates }) => client.updateTask(id, updates),
});
```

## How strategies interplay

- **Per partition, not per app.** Each partition entry picks its own strategy — one collection can drain its `orgId` partitions `oldToNew` while another mirrors channels `newToOld`.
- **Lifecycle is demand-driven.** A partition's strategy starts on the first matching `loadSubset` and stops when the last consumer leaves; its Sync State is kept, so re-entering the partition resumes instead of restarting.
- **Restart-safe by contract.** Strategies never catch `WriteError` — the engine logs and restarts the run on a spaced schedule, and the persisted cursor guarantees the retry resumes correctly.
- **Cadence repair reuses `forwardFetch`.** The optional cadence loop periodically re-walks the forward direction to repair anything a subscription missed — same fetch source, no extra wiring.
