You've read the mechanics and the best practices, and if you're the kind of engineer we're writing for, you're now generating "okay, but what happens if…" scenarios faster than any tutorial covers them. Good — this page is that list. Every answer below is verified against the eschema source and its test suite, not against how we'd *like* it to behave. Examples use the chat app's `Message` schema: v1 is `{ channelId, authorId, body }`, v2 added `reactions`.

## What happens if I modify v1 after data exists?

Nothing detects the edit — and that's the danger. There is no registry of historical shapes beyond your source code, so decode validates every `_v: 'v1'` row (and every unstamped legacy row, which counts as v1) against whatever your v1 fields say *now*. If you renamed `body` to `text` in v1, every old message fails at read time with a `Decode failed` error — loudly, in production, on rows that were fine yesterday. And if the edited v1 happens to still accept old rows, you get something worse: silent reinterpretation.

The library's contract is that shipped versions are frozen; the test suite pins the failure mode (an unstamped row that doesn't match v1 rejects rather than guessing another version). The [CLI](/docs/std-toolkit/eschema/cli) turns the contract into a CI failure by snapshotting each version's source and flagging any drift.

## What if decode meets a `_v` the schema doesn't know?

Say a newer deployment is writing `_v: 'v3'` messages while an older service still runs a schema whose chain ends at v2. The old service's decode fails with `ESchemaError: Unknown schema version: v3`. There is no forward migration and no "best effort" pass-through — eschema can fold old data forward, never new data backward.

That's a deliberate refusal, not a gap: the v2 code has no idea what v3 means, and pretending otherwise corrupts data. The operational consequence is a deploy-ordering rule — see [the two-services question](#what-if-two-deployed-services-run-different-schema-versions) below.

## What if a row has no `_v` at all?

It decodes as the *earliest* version — v1 — and folds forward through the whole chain. A pre-eschema message `{ id, channelId, authorId, body }` decodes against v1's fields, then the v2 migration adds `reactions: {}`, and your code sees a current-shape message. This is the rule that makes adoption over existing data non-breaking, and it's pinned by tests at every level: unstamped top-level rows, unstamped values nested inside stamped parents, even arrays mixing unstamped and stamped elements — each folds forward through its own chain independently.

The contract you owe in return: your v1 must faithfully mirror what your legacy data actually looks like. An unstamped row that doesn't match v1 fails loudly — eschema never guesses a different version.

## Can I ever delete a version?

No — not while any row stamped with it might still exist, and in practice you can't prove that negative. Deleting a version breaks two ways:

- Rows stamped `_v: 'v2'` now hit a chain with no v2 entry: `Unknown schema version: v2`.
- Rows stamped *earlier* than the deleted version break too, because folding runs through every intermediate step — v1 rows can't reach v3 without the v1→v2→v3 chain intact.

The version chain is append-only by design: `evolve` only accepts the next sequential version, so there's no renumbering around a gap. If you delete a `versions/vN.ts` file in a CLI-managed schema collection, `eschema lint` reports it as an un-approvable issue that must be fixed by restoring the file. The cost model is honest: a version is a few lines of schema plus a pure function, kept forever. That's cheap. Unreadable rows are not.

## What if two deployed services run different schema versions?

The compatibility is one-directional, and it's worth internalizing which direction:

- **New code reading old data: always fine.** That's the whole product — v1 rows fold forward to whatever the reader's latest is.
- **Old code reading new data: fails.** The v2 service meeting a `_v: 'v3'` row gets `Unknown schema version: v3`.

So during a rolling deploy, the moment one instance starts *writing* v3, every instance still *reading* at v2 can error on fresh rows. The rule that follows: **ship the schema change to all readers before anything writes it.** Deploy the new version chain everywhere (readers now understand v3 but nothing emits it yet, since encode only runs where new code writes), then let writers roll. In a monolith this is automatic; across services, the schema package must reach every consumer first. The same logic covers a mid-rollback window — anything already written at v3 will fail to decode on v2 code, which is a reason to keep migrations flowing forward rather than rolling schema versions back.

## What if a migration references a field the previous version doesn't guarantee?

It doesn't compile. The migration's `prev` parameter is typed as the previous version's decoded shape — reference `prev.reactions` in the v1→v2 migration that's *introducing* reactions, and TypeScript rejects it. Same for the output: the migration's return type is checked against the merged v2 shape, so forgetting to supply a new field is also a compile error.

And the runtime backs the types up: before your migration runs, the row has already been validated against the previous version's schema. `prev` is never malformed, never partially decoded — if the row doesn't match the old shape, decode fails *before* your migration, with the schema issue attached. Inside a migration you can lean on the old shape completely.

## What does `ESchemaError` carry and when is it raised?

It's a `Data.TaggedError('ESchemaError')` with three fields: a human-readable `message`, an optional `data` payload, and an optional `cause` carrying the underlying failure (for schema validation failures, that's the full Effect Schema issue with per-field details). Being a tagged error in the Effect error channel, it's typed on every signature and catchable by tag:

```ts
Message.decode(row).pipe(
  Effect.catchTag('ESchemaError', (err) =>
    Effect.logError('undecodable message row', { message: err.message, cause: err.cause }),
  ),
);
```

You'll see it in four situations: `Unknown schema version: <v>` (the row's stamp isn't in the chain), `Decode failed` (the row doesn't match its version's schema — `cause` has the details), `Encode failed` (the value you're writing doesn't match the latest shape), and `No evolutions found` (a degenerate schema with no versions — you'd have to construct one manually to see it).

## What if the value I pass to encode still has an old `_v` on it?

It's ignored and overwritten. Encode validates the value against the latest fields and stamps the latest version, unconditionally — there's a test feeding `{ _v: 'old', ... }` into encode and asserting the output carries the current version. You cannot smuggle an old version stamp through the write path.

## Does decoding an old row rewrite it in the database?

No. Migrations run in memory, on read, and the stored bytes are untouched — a v1 message can stay a v1 message on disk for years while every read sees the v2 shape. Rows upgrade physically only when something writes them again, because encode always emits the latest version. If you want to proactively rewrite old rows, that's an ordinary read-decode-encode-write loop you run on your own schedule — eschema makes it safe but never forces it.

## Can I skip versions when evolving?

No — `evolve`'s version parameter is typed as exactly the successor of the current version, so `v1` can only evolve to `'v2'`. Skipping to `'v5'` is a compile error. This isn't pedantry: the fold assumes a contiguous chain, and every stored row's stamp must land on a real link in it.

## Is anything different for `ValueESchema`?

The semantics match, but envelope detection adds two edges the object constructs don't have, both pinned by tests:

- `{ value: 'hello' }` **without** `_v` is *not* treated as an unstamped envelope. It's a bare legacy value like any other — so it's validated against v1's value schema as-is (and fails if v1 expects a plain string, since the input is an object).
- An object **with** a top-level `_v` is always read as an envelope — even if that `_v` is your own business field. `{ _v: 'business-version', value: … }` fails with `Unknown schema version: business-version`. If your domain value naturally carries a `_v` key, rename it or nest it a level deeper; see [best practices](/docs/std-toolkit/eschema/best-practices#envelope-shaped-values).

## A nested schema evolved but the parent didn't — what do stored parents decode to?

The new nested shape, everywhere, immediately. Nested eschemas embedded via `toSchema` decode through their own chains, so evolving `Attachment` to v2 changes what every stored `Message` decodes to — while every message row's own `_v` stays exactly where it was. There's a test asserting precisely this: same bytes, parent still v1, child value comes out folded to its latest. Intended behavior, but it's the one place local reasoning fails — the [non-isolation section](/docs/std-toolkit/eschema/best-practices#nesting-is-not-isolated) of best practices is about living with it.
