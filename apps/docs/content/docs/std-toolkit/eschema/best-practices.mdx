You know how this goes with any tool that touches persisted data: the API is easy, and the scars come from usage patterns nobody wrote down. This page writes them down. Every rule here exists because of a specific way eschema can bite, and each comes with the reason — none of them are style preferences.

## v1 is frozen. So is every other shipped version

The single most important rule. Once data exists under a version, that version's fields must never change — and v1 doubly so, because **unstamped legacy rows decode as v1**. Edit v1's shape and you've silently redefined what every pre-eschema row means; if an old row no longer matches, decode fails loudly at read time, in production, long after the edit looked harmless in review.

The discipline: a version, once deployed, is history. Want a different shape? That's what `.evolve` is for — append, never rewrite. This rule is important enough that the [eschema CLI](/docs/std-toolkit/eschema/cli) exists largely to enforce it mechanically: it snapshots each version's source and fails CI when a shipped version's file changes.

## Migrations must be total — and pure

A migration is `(prev) => next` for *every possible* `prev`. Not "every prev I expect" — every value the previous version's schema admits. eschema validates the row against the old schema before your migration runs, so `prev` is never malformed; but within that shape, no throwing, no `undefined` returns, no "this can't happen" branches. If an old row exists for which you have no answer, your schema change isn't ready.

Purity is the other half. Migrations run lazily, on read, at any time, any number of times — the same v1 row might migrate today on one server and next year on another. Anything nondeterministic (clock, randomness, IO, network) means the same stored bytes decode to different values on different days, which quietly breaks every layer above that assumes decode is a function.

```ts
// ✗ not total, not pure
.evolve('v2', { editedAt: Schema.NullOr(Schema.Number) }, (prev) => ({
  ...prev,
  editedAt: Date.now(),        // decodes differently every read
}))

// ✓ total and pure
.evolve('v2', { editedAt: Schema.NullOr(Schema.Number) }, (prev) => ({
  ...prev,
  editedAt: null,              // old messages were never edited — say so
}))
```

## Prefer `Schema.NullOr` over optional fields

When adding a field, model "no value" as `null`, not as an absent key. Optional fields poison migrations: if `editedAt` may or may not exist, every future migration touching it branches on `undefined`, and the encoded shape becomes unstable — sometimes the key is there, sometimes not. With `NullOr` the field is *always present* with a definite value: migrations stay total by construction, and the wire shape is predictable.

```ts
const Message = EntityESchema.make('Message', 'messageId', {
  channelId: Schema.String,
  body: Schema.String,
})
  .evolve('v2', { editedAt: Schema.NullOr(Schema.Number) }, (prev) => ({
    ...prev,
    editedAt: null, // explicit absence, not a missing key
  }))
  .build();
```

## One small step per version

Each migration handles exactly the immediately previous version — eschema chains them (v1→v2→v3→…), so a very old row folds through every step. Resist the urge to write one clever migration that inspects the data and handles "any old shape": that's reimplementing the version chain, badly, inside a function whose input type says otherwise. Small steps compose; clever steps rot.

The version names enforce half of this for you — `evolve` only accepts the next version in sequence, so you can't skip from v2 to v5 even by accident.

## Nesting is not isolated

The one genuinely surprising rule, and it follows directly from how [`toSchema`](/docs/std-toolkit/eschema/constructs#toschema--the-bridge-into-plain-effect-schema) works: every nested eschema decodes through its **own** chain. So when you evolve a nested schema, every parent embedding it starts decoding to a different shape — *without any parent version changing*.

Concretely: `Message` embeds `Attachment`, both at v1. You evolve `Attachment` to v2, adding `sizeBytes`. Every stored message — still stamped `_v: 'v1'`, byte-for-byte unchanged — now decodes with `sizeBytes` inside its attachments. That's the intended behavior (nested values migrate independently; it's what makes composition work at all), but it means the "nothing changes until I evolve" guarantee is a statement about the whole nesting tree, not one schema. **When you evolve a child, audit every parent that embeds it** — their decoded types just changed, and so did what their consumers see.

## Underscore-prefixed field names are reserved

`_v` and the rest of the `_` namespace belong to the library. The type system rejects `_`-prefixed keys in `make` and `evolve` deltas, so this one can't get past the compiler — it's listed here so the error message doesn't surprise you. (Values *inside* a `ValueESchema` envelope are exempt: the envelope isolates them, so `{ _source: 'import', name: 'Alice' }` is a fine enveloped value.)

## Envelope-shaped values

A `ValueESchema` decides "is this input an envelope or a bare legacy value?" by one signal: does the object have a `_v` key? That heuristic has a corner: if your *domain value itself* is an object with a top-level `_v` field, eschema will read your business data as a version stamp and fail with `Unknown schema version`. If a value you're versioning naturally carries a `_v`-like field, rename it in the value schema or wrap it one level deeper — don't fight the envelope.

## Schemas are immutable after `build` — and mildly lazy before use

Two mechanical facts that shape where schema code lives:

- `build()` returns a frozen instance. There's no appending versions later, so declare the entire chain in one place — the schema file *is* the changelog.
- Reading `.fields` or `.schema` during module initialization can race the schema's own setup and throws a descriptive error. Build schemas at module scope, but compute derived things inside functions, not in top-level expressions.

## The checklist

- Never modify a shipped version; only `.evolve`. Let [the CLI](/docs/std-toolkit/eschema/cli) enforce it in CI.
- Migrations: total, pure, one step, no IO — backfill *meaning*, not timestamps.
- New fields are `Schema.NullOr(...)` defaulting to `null`, not optional keys.
- Evolving a nested schema? Audit every parent that embeds it.
- Keep `_`-prefixed keys out of field maps, and top-level `_v` out of `ValueESchema` values.
- Build whole chains at module scope; derive lazily.

When one of these rules meets a "but what actually happens if…" question, the [Q&A page](/docs/std-toolkit/eschema/qa) answers it from source.
