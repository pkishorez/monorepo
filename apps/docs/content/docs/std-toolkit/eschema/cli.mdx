---
title: Snapshots
description: Mechanically enforce frozen versions with std-toolkit snapshot — a semantic contract baseline that fails CI when shipped history changes.
---

The [best practices](/docs/std-toolkit/eschema/best-practices) lean on one rule above all others: **a shipped version never changes.** And you already know what happens to rules that live only in prose — someone "just tidies up" a schema during a refactor, review doesn't catch it because the diff looks cosmetic, and three weeks later old messages stop decoding. Rules that matter need to fail CI.

That's what `std-toolkit snapshot` is for. It ships as the `std-toolkit` bin, and it does one thing: compare your schemas' **semantic contract** against an approved baseline, and fail when shipped history changed.

Semantic is the operative word. The snapshot isn't a copy of your source files — it's the derived contract: for every schema, every version's encoded and decoded shape; for a table, additionally its indexes and registered entities. Rename a variable, reorder fields, reformat the file — the snapshot doesn't move. Change what `v1` actually accepts, and it does.

## Setup

Create a `std-toolkit.snapshot.ts` at your project root that default-exports the snapshot of the thing you want under contract. For a table-based app, that's the table — its snapshot includes every registered entity and all their schema versions:

```ts title="std-toolkit.snapshot.ts"
import { table } from './src/db/table.js';

export default table.snapshot();
```

For standalone schemas, export the schema's own snapshot — `export default Message.snapshot()`. Either way, nested schemas composed via `toSchema` are pulled in automatically; the contract covers the whole tree.

## The command

```sh
std-toolkit snapshot        # verify against the approved baseline
std-toolkit snapshot -u     # approve: write the current contract as the baseline
```

The first run has no baseline, so it prints the rendered contract and asks you to approve it. `-u` writes `std-toolkit.snapshot.json` — that file is generated output, never edited by hand, and it goes into the same commit as the schema change. From then on, every plain run diffs the current contract against it: exit `0` when nothing changed, exit `1` when something did.

Run it in CI next to your type check:

```yaml title=".github/workflows/ci.yml"
- run: npx std-toolkit snapshot
```

With that line, "never modify a shipped version" stops being a code-review hope and becomes a red ✗ on the pull request.

## Reading the report

Every detected change is classified by what it means for data already in storage:

- **safe** — appending: a new version, a new entity, a newly composed schema. This is the normal, encouraged operation; approve and move on.
- **requires-backfill** — the contract grew in a way existing rows don't satisfy yet, like a new secondary index. Nothing breaks, but the index is only as good as the data behind it.
- **breaking** — shipped history changed: a version was modified or deleted, an entity or schema removed, an id field changed. This is the drift the whole system exists to catch. Approving one should be as loud in review as the word sounds.
- **unverifiable** — the change involves behavior the snapshot cannot see inside, such as a custom transformation. The tool also prints warnings for these on every run: it can prove the shapes match, but not what the code does between them.

A `-u` run records whatever the current contract is — the classifications exist to make sure a human looked first.

## The loop, end to end

Day one: write your schemas, point `std-toolkit.snapshot.ts` at them, `std-toolkit snapshot -u`, commit source and baseline together. Months later, reactions ship: add the `.evolve('v2', ...)` step, run `std-toolkit snapshot` — it reports a safe `version-added` — approve with `-u`, commit both. In between, every PR runs the plain command, and the one PR that accidentally edits `v1` gets stopped by a machine instead of a reviewer's tired eyes.

One file in, one file out, one command wrapping one idea: the version history is a contract, and contracts get diffed, not trusted.
