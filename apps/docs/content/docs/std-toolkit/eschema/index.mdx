You already validate data at the boundary. A schema library sits at the edge of your app, rejects malformed input, and hands you typed values — that problem is solved, and you've solved it many times.

Here's the problem that isn't solved: **your code is a point in time, but your data is forever.**

Take the chat app. Version one of a message is honest and small:

```ts
{ channelId: 'general', authorId: 'alice', body: 'ship it' }
```

Six months later, product wants reactions. Your *code* changes in one deploy — but the million messages already sitting in the database were written by the old code, and redeploying your app does not rewrite rows. The moment `reactions` becomes part of the message shape, every one of those old rows is invalid under your new schema.

You know the usual ways out, and none of them feel great:

- **The big-bang backfill.** Write a migration script, run it over every message in production, hope it finishes, hope it's right, coordinate it with the deploy. Now do that for every schema change, forever, on every database your data lives in.
- **Optional-field soup.** Make `reactions` optional. And the next field. And the one after that. Two years in, your "schema" is a pile of `?`s, and every reader everywhere branches on which fields happen to exist.
- **Hand-rolled version checks.** Store a `version` column and `if`/`else` your way through history at every read site. It works — it's also exactly the kind of mechanical, error-prone bookkeeping a library should own.

## eschema's answer

An eschema is a schema that knows its own history. It's a chain of versions — `v1`, `v2`, … latest — where each step records *what changed* and *how to fill the gap* for older data. Two rules run the whole show:

- **`encode` always writes the latest version** — and stamps a `_v` field recording it. There is no way to write an old shape; old shapes exist only to be read.
- **`decode` reads `_v` and folds forward.** A row stamped `v1` is validated against the v1 shape, then migrated v1 → v2 → … → latest, one small step at a time. Your code only ever sees the current shape.

Here's the message schema, before and after reactions ship:

```ts title="message-schema.ts"
import { Schema } from 'effect';
import { EntityESchema } from 'std-toolkit/eschema';

export const Message = EntityESchema.make('Message', 'messageId', {
  channelId: Schema.String,
  authorId: Schema.String,
  body: Schema.String,
})
  .evolve(
    'v2',
    { reactions: Schema.Record(Schema.String, Schema.Number) },
    (prev) => ({ ...prev, reactions: {} }),
  )
  .build();
```

And here's what the two rules buy you:

```ts
Effect.gen(function* () {
  // A row written last year, before reactions existed.
  const oldRow = {
    _v: 'v1',
    messageId: 'msg-1',
    channelId: 'general',
    authorId: 'alice',
    body: 'ship it',
  };

  const message = yield* Message.decode(oldRow);
  // => { messageId: 'msg-1', channelId: 'general', authorId: 'alice',
  //      body: 'ship it', reactions: {} }
  // The migration ran on read. The database row is untouched.

  const encoded = yield* Message.encode(message);
  // => { ..., reactions: {}, _v: 'v2' } — always the latest version.
});
```

No backfill script ran. No reader branches on versions. The migration is written once, next to the schema change it belongs to, and it runs lazily on every read of an old row.

## The adoption question you're already asking

"Fine for new projects — but my database is full of rows with no `_v` at all." That case is designed in: **a row with no `_v` decodes as v1.** Wrap your existing schema as the v1 of an eschema and nothing changes — on a single-version schema, stamped and unstamped rows decode identically. The rule only starts paying off later, when you evolve past v1 and your legacy rows fold forward like everything else. Adopting eschema over existing data is a non-breaking change.

## Why this sits under everything else

This isn't a convenience feature bolted onto the toolkit — it's a load-bearing wall. Every persisted [Entity](/docs/std-toolkit/core) in std-toolkit carries `_v` in its [metadata](/docs/std-toolkit/core/meta), and that `_v` is an eschema version. The [database adapters](/docs/std-toolkit/database-adapters) store what eschemas encode; the sync engine ships rows of mixed vintages to browsers running the newest code and trusts decode to reconcile them. A sync engine that moves rows written across months of schema changes only works if schema evolution is solved first. This section is where we solve it.
