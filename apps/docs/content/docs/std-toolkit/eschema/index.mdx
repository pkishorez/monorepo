## Persistence changes everything

Persist some data — a file, a database row, a record in any store — and that data needs a schema. Without one, nothing tells you what shape comes back out. So you add a schema, and safety returns.

But requirements change, and sooner or later the schema is modified. What happens then depends entirely on *where* the old shape lives.

## Point-in-time structures are fine

For API payloads, the structure is **point in time**: a request comes in, a response goes out, and nothing is kept. Change the structure and the end consumers simply adapt — regenerate a client, update a caller — and everything keeps working. There is no history to worry about, because there is no history at all.

## Persisted structures are not

Persisted data is different: every row ever written is history, frozen in the shape the schema had on the day it was written. Ship a breaking change to that schema and the data already on disk is fundamentally out of step — the new schema can no longer read it. There is no going back in time to rewrite what was persisted, and shuttling millions of rows back and forth between shapes with migration scripts is tedious, risky, and often needs downtime.

So: a schema is a great abstraction for a **point-in-time** definition of data. For data that must outlive changing requirements, a schema alone is the wrong abstraction.

## Enter the evolving schema

That is where the evolving schema comes in. An **eschema** defines not only the shape of the data, but **how that shape evolves over time**: a chain of versions `v1 … latest`, where each step carries a small pure migration from the previous shape. Old data is never rewritten in place — it is **folded forward on read**.

Decode reads the `_v` stamp on each row, finds that version in the chain, and applies each migration in order until the value reaches the latest shape. Encode always writes the latest version. Old versions are only ever read, never written.

## Three scenarios

**A field is added.** Requirements say users now have an email. With a plain schema, every existing row is suddenly invalid. With an eschema, you add one `.evolve('v2', …)` step whose migration defaults the new field — old rows decode as v1, fold forward, and arrive with `email: null`. No script touched the table.

**A field is renamed.** `name` should really be `fullName`. One evolve step removes the old key, adds the new one, and the migration copies the value across. Rows written years apart all decode to the same latest shape.

**Old, unstamped data.** Data written before eschema existed carries no `_v` at all. It decodes as `v1` — so wrapping an existing store's current shape as v1 is a pure no-op, and evolutions can start from there. Adoption is non-breaking by design.

In every scenario the payoff is the same: **no migration scripts, no downtime, no shape-branching at read sites.** Your application code sees exactly one shape — the latest — and each migration is one small pure function from exactly the previous version to the next.
