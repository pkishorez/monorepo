## The story

Schemaless stores like DynamoDB make writing easy: any shape goes in. The cost shows up later, when nothing tells you what shape comes *out*. So you add a schema — and safety returns, but a new problem arrives with it: **evolution**. A schema pins your data's structure in place. The day you need to add a field, rename one, or change a type, you face the migration problem: millions of rows written in the old shape, and a schema that only understands the new one.

The classic answers are all painful. Migrate the whole table in one pass (downtime, risk). Handle every historical shape at every read site (branching code that grows forever). Or give up and make every field optional (safety quietly leaves again).

eschema takes a different path: **encode the evolutions inside the schema itself.** A schema is not a snapshot — it is a chain of versions, `v1 … latest`, where each step carries a small pure migration from the previous shape. Old data is never rewritten in place; it is **folded forward on read**.

Decode reads the `_v` stamp on each row, finds that version in the chain, and applies each migration in order until the value reaches the latest shape. Encode always writes the latest version. Old versions are only ever read, never written.

## Why this wins

- **No migration scripts, no downtime.** Rows migrate lazily, one at a time, at the moment they are read.
- **No shape-branching at read sites.** Your application code sees exactly one shape: the latest.
- **Adoption is non-breaking.** Data with no `_v` stamp decodes as `v1`, so wrapping an existing table's shape as v1 is a pure no-op — evolutions come later.
- **Each migration is small.** One pure function per version step, from exactly the previous shape to the next. The chain composes; you never write "handle any old version" code.
