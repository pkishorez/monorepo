We started with DynamoDB's way of seeing data: one table made useful by partition keys, sort keys, and direct access paths.

Then std-toolkit tightened that model into typed item collections. Each collection has one entity type, one stable order, and a cursor a client can carry with it.

Now we can close the loop by following one message.

User1 opens `#general` and types **“Seen the logs?”** User2 already has the same channel open in another browser. For User1's sentence to appear on User2's screen, every layer adds one small piece of meaning.

## The message begins as a value

To the application, the message is ordinary domain data: a channel, an author, and some text.

An [eschema](/docs/std-toolkit/eschema) defines that shape and its identity. It also remembers how the shape evolved.

```ts title="message-schema.ts"
import { Schema } from 'effect';
import { EntityESchema } from 'std-toolkit/eschema';

export const Message = EntityESchema.make('Message', 'messageId', {
  channelId: Schema.String,
  authorId: Schema.String,
  body: Schema.String,
})
  .evolve(
    'v2',
    { reactions: Schema.Record({ key: Schema.String, value: Schema.Number }) },
    (previous) => ({ ...previous, reactions: {} }),
  )
  .build();
```

The schema says this is a `Message`, that `messageId` identifies it, and that a message written before reactions existed should still decode into today's shape.

This is the first guarantee: no matter when a row was written, the rest of the application receives a valid current message.

## The adapter gives it an address

The database adapter connects the message schema to the item-collection model.

```ts title="message-entity.ts"
const messages = DynamoEntity.make(table)
  .eschema(Message)
  .primary({ pk: ['channelId'] })
  .build();
```

From that declaration, std-toolkit derives the keys:

```text
pk = Message#<channelId>
sk = <messageId>
```

User1's new message is validated and inserted:

```ts title="send-message.ts"
const sent =
  yield *
  messages.insert({
    messageId: yield * nextUlid,
    channelId: 'general',
    authorId: 'user1',
    body: 'Seen the logs?',
    reactions: {},
  });
```

The adapter turns the access pattern into a concrete address:

```text
pk = Message#general
sk = 01J...A2
```

The message now lives in the same physical table as channels and members, but it sits in the typed collection for `#general` messages. Its ULID places it after the messages created before it.

The DynamoDB mental model has become application structure: the screen's question, “messages in this channel,” is encoded directly in the key.

## The value gets the context sync needs

A valid value is enough to render, but it is not enough to synchronize. A client also needs to know which entity type it received, which schema version wrote it, whether it was deleted, and which copy is newer.

std-toolkit carries those answers beside the value in an entity envelope:

```ts
{
  value: {
    messageId: '01J...A2',
    channelId: 'general',
    authorId: 'user1',
    body: 'Seen the logs?',
    reactions: {},
  },
  meta: {
    _e: 'Message',
    _v: 'v2',
    _d: false,
    _u: '01J...U7',
  },
}
```

`_e` names the entity type. `_v` records the schema version. `_d` marks a tombstone. `_u` is the fresh update key assigned to this write.

That update key gives every copy a comparable age. When two envelopes for the same message meet, the one with the larger `_u` wins.

Duplicates become harmless. An older response arriving late cannot overwrite a newer edit. An old live message cannot resurrect a newer tombstone.

This is where the two collection orders meet: `messageId` places the message in chat history, while `_u` places this write in change history.

## User2 continues from their cursor

User2's browser does not ask for the whole table. Their open channel activates the sync work for `Message#general`.

The browser reads its saved cursor and asks for the next change-ordered page. The server returns envelopes after that position, including User1's message, and the browser advances the cursor.

```text
Message#general changes:  A  B  C  User1's message
User2 has:                A  B  C
saved cursor:                   C
next page:                          User1's message
```

If User2 reloads before the message arrives, the browser can first render its persisted local data and then resume from the same cursor. The server never has to remember where User2 stopped.

Pull is not the only way the envelope may arrive. A live transport can push it immediately as well. Both paths feed the same local truth and use the same `_u` comparison, so their arrival order does not matter.

The cursor answers “what should I request next?” The update key answers “should this copy replace the one I have?” Together they make movement resumable and convergence deterministic.

## TanStack DB turns collections back into a screen

The server deliberately sends typed collections rather than a pre-joined channel page. On User2's device, TanStack DB composes the local channel, message, and member collections into the view they need.

The messages themselves are a normal live query:

```tsx title="channel-view.tsx"
const { data: messages } = useLiveQuery((query) =>
  query
    .from({ message: messagesCollection })
    .where(({ message }) => eq(message.channelId, channelId)),
);
```

When User1's envelope converges into the local message collection, this query updates and User2 sees **“Seen the logs?”**

The server did not join the channel, its members, and its messages. It kept each read typed and predictable. The client combined those local pieces into the application screen.

This closes the first half of the story: **the server stores and moves entities; the client composes them into experiences.**

## User2's reply travels the same path back

User2 types **“Looking now.”** TanStack DB applies the insert optimistically, so their own reply appears without waiting for a round trip.

The mutation sends the value to the server. The schema validates it, the adapter places it in `Message#general`, and the server returns a confirmed envelope with its real metadata and update key.

That envelope converges over the optimistic copy. A live broadcast may deliver the same confirmed write again, but the duplicate `_u` makes it a no-op. User1's browser receives the new change after its own cursor and renders User2's reply through the same local query.

The path is symmetrical. Local interaction becomes a typed value, the value receives a stable address and sync metadata, the envelope moves through an ordered collection, and another local database turns it back into a reactive screen.

## The whole model

Each layer solves one problem and hands a small, stable shape to the next.

| Layer                  | What it contributes                                   |
| ---------------------- | ----------------------------------------------------- |
| **eschema**            | Current shape, identity, validation, and migrations   |
| **core**               | Entity metadata, tombstones, and convergence by `_u`  |
| **database adapter**   | Keys, typed operations, and one-table storage         |
| **item collection**    | A bounded group with a stable, resumable order        |
| **tanstack-sync**      | Cursors, local truth, offline recovery, and mutations |
| **TanStack DB and UI** | Local joins, live queries, and optimistic interaction |

DynamoDB supplied the organizing idea: design direct access paths as ordered collections. std-toolkit made those collections typed and portable across databases. Cursors made them resumable. Entity envelopes made concurrent delivery converge. TanStack DB recomposed them into a local-first application.

The loop is now complete:

> **One table holds the entities. Item collections make them addressable and syncable. Cursors move through them. The client turns them back into an application.**
