---

DynamoDB can feel unfamiliar if you come from a relational database. There are no joins, and you do not begin by making a table for every kind of data.

You begin with the questions your application needs to answer.

For a chat app, those questions might be “give me the messages in `#general`” or “give me the latest 50 messages in that channel.” DynamoDB asks you to organize the data so those reads already have a direct path.

That shift—from modeling entities first to modeling access patterns first—is the most important part of the DynamoDB mental model.

## One table can hold the application

In a relational database, channels, messages, and members would usually live in separate tables. In DynamoDB single-table design, they can share one physical table.

That does not make the table one giant unordered bucket. Every item has keys that place it into a small, ordered group.

| pk                | sk         | entity  | body             |
| ----------------- | ---------- | ------- | ---------------- |
| `Message#general` | `01J...A1` | Message | `Morning!`       |
| `Message#general` | `01J...A2` | Message | `Seen the logs?` |
| `Message#random`  | `01J...B1` | Message | `Lunch?`         |
| `Channel`         | `general`  | Channel | —                |
| `Channel`         | `random`   | Channel | —                |

The rows share a table, but their keys give them structure. The table is only the physical home. The keys describe how the application reaches the data inside it.

## The partition key chooses the group

The **partition key**, or `pk`, answers one question:

> Which items belong together?

A chat screen reads messages one channel at a time, so a useful partition key includes the channel id:

```text
Message#general
Message#random
```

Every message with `pk = Message#general` belongs to the same logical group. A DynamoDB Query supplies that exact partition key and goes directly to the group instead of searching the table.

DynamoDB also uses the partition key to distribute data across its storage. Different partition-key values can be stored and served independently, which is how a single table can grow without every read touching all of it.

This gives us the first design rule:

> Put together the items your application reads together.

## The sort key gives the group an order

The **sort key**, or `sk`, answers the next question:

> Where does this item sit inside its group?

std-toolkit uses an entity's id as its primary sort key. For timeline data, a time-sortable ULID makes creation order and text order agree.

```text
Message#general
├── 01J...A1  Morning!
├── 01J...A2  Seen the logs?
└── 01J...A3  Looking now.
```

Once the items have a stable order, familiar reads become sorted ranges. You can start at the beginning, read backward from the end, or continue after the last key you saw.

The sort key is therefore more than uniqueness. It is the small query language inside a partition.

## Together, the keys make an item collection

DynamoDB calls the items with one partition-key value an **item collection**. Its items are read in sort-key order.

The table may contain millions of items, but an application usually works with a much smaller collection such as `Message#general`. This is the level at which a read becomes predictable: choose one collection, choose a position or range, and choose a limit.

That is the central idea std-toolkit borrows from DynamoDB.

## Query follows the model

A **Query** reads one item collection, or a sorted range within it. It follows the path already encoded in the keys.

A **Scan** walks items across the table. Its cost grows with all the data it examines, including data unrelated to the question.

For “messages in `#general`,” a Query goes directly to `Message#general`. A Scan would also walk messages from `#random`, channels, members, and anything else in the table.

If an important read requires a Scan, that usually means the keys do not represent the access pattern yet.

## Indexes add another access path

One pair of keys gives an item one useful grouping and order. Real applications often need more than one.

The primary keys might answer:

> Which messages were created in this channel, and in what order?

Sync needs a slightly different question:

> Which messages in this channel changed after my last cursor?

A secondary index gives the same items another set of keys:

```text
primary access path
pk = Message#general
sk = messageId

change access path
pk = Message#byChannel#general
sk = _u
```

The first path orders messages by id, which is useful for browsing chat history. The second orders them by std-toolkit's `_u` update key, which changes on inserts, edits, and tombstones. That makes it useful for sync catch-up.

The item is not a different domain object in each index. An index simply makes the same data efficiently queryable through another grouping or order.

### GSI and LSI, briefly

A **global secondary index**, or GSI, can introduce both a new partition key and a new sort key. DynamoDB updates it asynchronously, so reads from a GSI are eventually consistent.

A **local secondary index**, or LSI, keeps the table's partition key and introduces another sort key. It must be declared when the table is created.

The details matter when operating DynamoDB, but the mental model is small: an index is another access path over the same items.

## Keep this picture

Start with the questions the application asks. Turn each important question into a direct access path. Use a partition key to select the group and a sort key to make the group navigable.

```text
one table
  → many partition-key groups
    → each group ordered by a sort key
      → Query reads a range from one group
        → indexes provide other useful groupings and orders
```

Capacity modes, billing, Streams, and provisioning all matter when running DynamoDB in production. They are not needed to understand what std-toolkit takes from it.

The idea we need is simply this: **one table can expose many small, ordered collections that are cheap to address and easy to continue through.**
