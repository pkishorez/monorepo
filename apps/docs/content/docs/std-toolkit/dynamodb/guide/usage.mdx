## Client and layers

The client is constructed with `createDynamoDB`, exposed as the `DynamoDB` Effect service, and provided through `dynamoDBLayer`. `DynamoConnection` describes the connection; failures surface as `DynamodbError`.

## Table and entity services

- `DynamoTable` — operations over the physical single table (the shared topology: partition key, sort key, item collections).
- `DynamoEntity` — the per-entity CRUD surface for keyed entities; `DynamoSingleEntity` for singletons.
- `EntityRegistry` — coordinates all entity types stored in the one table, routing operations to the right entity by its type tag.

Every persisted record is a core Entity (`{ value, meta }`); the `value` is validated by the entity's eschema schema on the way in and folded forward on the way out.

## Expression builders

`exprCondition`, `exprFilter`, and `exprUpdate` build type-safe condition, filter, and update expressions; `buildExpr` assembles them into the SDK's expression/attribute-name/attribute-value triple. `opAdd` and `opIfNotExists` cover the update-operation forms (`ADD`, `if_not_exists`).

## Marshall

`marshall` and `unmarshall` convert between plain values and DynamoDB attribute values.

## Migration

`MigrationOptions` and `MigrationReport` support table migration runs, reporting what was scanned and rewritten.

## RPC

RPC helpers for exposing table operations over the wire live under the `std-toolkit/dynamodb/rpc` subpath. Advanced internal paths are importable via the `std-toolkit/dynamodb/*` wildcard.
